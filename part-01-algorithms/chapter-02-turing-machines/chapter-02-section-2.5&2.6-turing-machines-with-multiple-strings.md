# Chapter 2 — Sections 2.5 & 2.6: Space Bounds and Random Access Machines

Space bounds require more care than time bounds to define correctly, and Section 2.5 is where that care is taken. The naive approach -- just measure the total length of all strings at the end of the computation -- has an obvious flaw: it overcharges every computation for reading its own input. If the input has length n, the input tape alone contributes n to the space count, making it impossible to study algorithms that use space genuinely smaller than n. But such algorithms exist and are important -- the palindrome example from Section 2.3 uses only O(log n) workspace beyond the input itself.

The fix is to separate the input tape (read-only) from the work tapes, and the output tape (write-only) from both. This is the *k-string Turing machine with input and output*, and the space charge counts only the work tapes, not the input or output. Proposition 2.2 confirms that this restriction doesn't cost anything computationally -- any machine can be simulated by one with this separation. The important space complexity class that emerges is L = SPACE(log n), the class of problems solvable in logarithmic workspace. The fact that REACHABILITY is already shown to be in L (or nearly -- the O(log² n) algorithm from Section 1.1 lands in SPACE(log² n), while the true result L ⊆ REACHABILITY ⊆ NL is developed in Chapter 16) is a preview of one of the book's more surprising results.

The relationship between time and space is one of the genuinely deep open questions in complexity theory. We know that TIME(f(n)) ⊆ SPACE(f(n)) -- a machine cannot use more space than it has time to visit -- and that SPACE(f(n)) ⊆ TIME(2^O(f(n))) -- a machine with f(n) space can only be in 2^O(f(n)) distinct configurations before it must repeat itself and loop. But whether P = L, or whether L = NL, remains open. These relationships are explored in depth in Chapters 7 and 16.

Section 2.6 brings in the Random Access Machine (RAM), and this is where the book makes its most direct argument that Turing machines are not an arbitrary formal choice but the *right* foundation for complexity theory. The RAM is a model that looks like an actual computer: it has an array of registers each capable of holding an arbitrarily large integer, an instruction set with arithmetic, conditionals, and jumps, and a program counter. This is essentially the idealized assembly language of a real machine. And yet -- Theorem 2.5 -- any RAM program running in time f(n) can be simulated by a Turing machine in time O(f(n)³). Polynomial in, polynomial out.

The proof of Theorem 2.5 is a beautiful piece of engineering. The Turing machine must represent the RAM's register file (which can be sparse and arbitrarily large) as a sequence of (register index, value) pairs on a tape. Each RAM instruction requires scanning this list, possibly updating an entry, and performing binary arithmetic. The cubic overhead comes from the fact that register values can grow to length O(f(n)) bits, and each arithmetic operation on such values takes O(f(n)) steps, done O(f(n)) times. A O(f(n)³) slowdown is polynomial -- and thus, for the purposes of complexity theory, irrelevant.

What this establishes is that when we say a problem is in P, we don't have to specify whether we mean "solvable by a Turing machine in polynomial time" or "solvable by a RAM program in polynomial time." The answer is the same either way. This gives the class P a kind of model-independence that is crucial for its role as the mathematical definition of tractability. It is worth pausing to appreciate this: we are not just saying the two models are equivalent in what they compute -- we knew that from Church's Thesis -- but that they are equivalent in *how efficiently* they compute it, up to polynomial factors. Cook and Reckhow made this precise in a foundational paper (Cook, S.A.; Reckhow, R.A. "Time bounded random access machines," *Journal of Computer and System Sciences*, 7(4), 1973), and Papadimitriou's treatment here is a self-contained version of that argument.

One subtlety in Section 2.6 deserves emphasis: the RAM model uses *unit cost* for arithmetic operations, meaning adding two billion-digit numbers counts as one step. This is obviously generous -- an actual computer would need time proportional to the number of digits. The justification for this liberality is Theorem 2.5 itself: even with this generous cost model, RAM programs are polynomially equivalent to Turing machines. If the RAM model were even more generous -- say, with a primitive multiplication instruction -- the equivalence would break down, and RAM programs could compute in polynomial time things that Turing machines seemingly cannot. The absence of MPLY from the basic instruction set in Figure 2.6 is therefore not an oversight but a deliberate boundary condition, and Example 2.7 shows that multiplication can be implemented efficiently within the restricted model anyway.
